internal/vm Split Plan

The current internal/vm package is carrying too many responsibilities: public VM facade, platform backend orchestration, managed guest policy, sidecar protocol, mount state, network state, built-in BSD launch policy, and large integration tests. This plan splits leaf subsystems first and keeps the top-level VM facade stable while the internals move into clearer packages.

Problem

internal/vm has become the place where cross-cutting runtime features land because it already imports the request types, image store, virtio filesystems, managed guest packages, and host-specific backends. That made sense incrementally, but it now obscures ownership and makes every new operating system look like another partial VMM living inside the same package.

Current shape63 Go files, roughly 12.7k lines in internal/vm.

The largest files are vm.go, sidecar_host.go, runtime_boot_test.go, and the platform backend files for Linux, Darwin, and Windows.

Design Rules

  • internal/vm should be the facade and orchestration layer, not the owner of every subsystem.
  • Extract leaf packages first to avoid circular imports and reduce review risk.
  • Package names should describe the domain directly. Avoid naming everything vm....
  • Keep client-facing APIs and runtime behavior stable during the split.
  • Move tests with the code they validate, and keep integration tests at the facade boundary.

Target Package Shape

internal/vm

Public facade, request routing, instance interface, backend selection, and high-level orchestration.

internal/vm/mounts

Runtime share state, image mount state, duplicate tracking, share conversion, and snapshot helpers.

internal/vm/netstate

Managed network state, guest IPv4 reporting, port forward wrappers, service proxy allowlist helpers.

internal/vm/execplan

Managed exec request resolution, control request validation, capability-gated feature errors, environment merging, rootdir handling, and user normalization hooks.

internal/vm/sidecar

Worker protocol, startup handshake, launch/dial helpers, sidecar managed session, and daemon lifecycle.

internal/vm/builtin

Built-in guest image lookup and launch descriptors for operating systems like OpenBSD and FreeBSD.

Dependency Direction

The split should keep dependencies flowing in one direction. Lower-level packages should not import internal/vm. 

client, imagefs, oci, virtio, vmruntime
        ↓
mounts, netstate, managed/execplan, sidecar protocol
        ↓
internal/vm facade and backend orchestration
        ↓
host-specific hv/kvm/hvf/whp implementation details

Phase 1: Extract Mount State

Move shares.go and managed_snapshot_helpers.go into internal/vm/mounts. This is the cleanest first step because the code mostly depends on client, oci, virtio, imagefs, and vmruntime.

Moves

  • managedMountState becomes mounts.State.
  • addRuntimeShareMount becomes a method or small package function.
  • addImageMount becomes a method or small package function.
  • Snapshot helpers move beside mount state because they operate on mounted roots/images.

Acceptance

  • No internal/vm/mounts import of internal/vm.
  • Existing share/image/snapshot behavior stays covered by moved tests.
  • internal/vm only calls the package instead of owning the state maps.

Phase 2: Extract Network State

Move shared network state and wrappers out of network_common.go into internal/vm/netstate. Leave platform-specific network runtime implementations in place initially.

Moves

  • managedNetworkState becomes netstate.State.
  • Port forward and service proxy wrapper helpers move with it.
  • IPv4 fallback/reporting logic moves with it.

Acceptance

  • Linux/Darwin/Windows network runtimes still satisfy small interfaces used by netstate.
  • No package cycle through internal/vm.
  • Tests validate user-visible behavior: disabled network errors, fallback behavior, and IPv4 reporting.

Phase 3: Extract Managed Exec Planning

Move the policy code for managed exec/control requests into internal/vm/execplan. This makes it clear that request normalization and capability checks are guest-management policy, not backend orchestration.

Moves

  • managed_exec_request.go
  • managed_control_request.go
  • managed_feature.go
  • Applicable helpers from managed_session_helpers.go

Acceptance

  • Capability gates are tested through allowed/denied operations, not table mirrors.
  • The package accepts interfaces/functions for path resolution and user lookup.
  • internal/vm composes the planner with roots, env defaults, and user resolution.

Phase 4: Extract Sidecar Protocol and Session

Move the sidecar worker protocol and managed session out before moving the entire sidecar host. This avoids a large tangled extraction and immediately shrinks sidecar_host.go.

Moves

  • worker_protocol.go
  • sidecar_managed_session.go
  • Startup hello parsing, worker dialing, launch command creation, daemon close/wait helpers.

Deferred

Keep sidecarVMHost orchestration in internal/vm until the protocol/session boundary is stable. The full host extraction can happen later.

Phase 5: Extract Built-in Guest Policy

Move built-in image lookup and guest launch descriptors into internal/vm/builtin. This package should describe what @openbsd and @freebsd mean without owning the low-level backend boot process.

Moves

  • Profile lookup for built-in BSD images.
  • Guest-specific hostname/interface/cache defaults.
  • Artifact builder descriptors for OpenBSD and FreeBSD, parameterized by network spec.

Acceptance

  • Adding another built-in OS does not require editing backend-specific routing tables in multiple places.
  • Backend code receives a launch descriptor and handles the host mechanics.

Phase 6: Backend Package Split

Split host backend implementations only after the leaf packages are extracted. Doing this first would be mostly mechanical churn and would likely create import cycles.

Possible End State

  • internal/vm/host/kvm
  • internal/vm/host/hvf
  • internal/vm/host/whp
  • internal/vm/host/sidecar

The exact names can be revisited when the boundaries are clearer. The important part is that host packages implement small backend interfaces and do not own guest policy.

Testing Strategy

  • Move unit tests with extracted code.
  • Keep runtime boot tests in internal/vm or a future integration package because they validate the facade.
  • Avoid tests that mirror private tables or exact helper construction.
  • Prefer tests that validate functional requirements: mount conflict behavior, capability gates, network fallback, worker handshake, and guest exec behavior.

Review Strategy

  1. One extraction per commit where possible.
  2. No behavior changes in extraction commits unless a bug falls out and is separately explained.
  3. Run go test ./internal/vm ./internal/managed/... ./internal/vm/mounts ./internal/vm/netstate ./internal/vm/sidecar as packages appear.
  4. Run go test ./... before each PR update.

Risks

  • Package cycles through client, vmruntime, or internal/vm.
  • Over-exporting internals just to make extraction easy.
  • Moving build-tagged files without preserving platform behavior.
  • Large sidecar extraction turning into behavioral changes.

First Concrete Step

Start with internal/vm/mounts. It has the best dependency shape, it removes real clutter from internal/vm, and it gives the refactor a low-risk first merge point.
